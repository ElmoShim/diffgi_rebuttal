## Garagenet 속도 관련


### 1. Geometry Image 표현공간
- 우리의 raw 표현공간 : 256x256x4
- GarmageNet 의 표현공간 : N x (256x256x4 + 8), N 평균 13, maximum 32
- 즉 GarmageNet 은 우리보다 (32)배 큰 표현공간을 사용해서 같은 의상 데이터를 표현하고 있음
- 실제 GarmageSet 데이터셋 평균 face 갯수는 141,064 개인데, diffusion 으로 생성 후 tesselation 하면 (약 220K ~ 1.05M)개짜리 엄청 dense 한 mesh 가 나옴 (diffgi paper or supp 참고)


### 2. latent vector (압축률)
- GarmageNet 은 32 x 72 latent diffusion
- 우리는 32x32x4 latent diffusion
- 우리의 Latent vector 크기가 GarmageNet 보다 1.78배 더 큼

### 3. 모델 사이즈
- GaramageNet Decoder : 약 2.42M params (AutoencoderKL, block_out_channels=[16,32,32,64,64,128], latent_channels=1, 256x256x4 → 8x8x1)
- GarmageNet Diffusion : 약 160M params (12 layers, hidden 768, num_heads 12, SpatialDiTBlock with cross-attn)
- ours Decoder : 약 49.5M params
- ours Diffusion : 약 458M params

- GarmageNet Inference : diffusion + 32 VAE decode
- ours Inference : diffusion + VAE Decode

### 결론
- GarmageNet 이 더 작고 단순한 벡터를 diffusion 함. 즉 더 빠름
- 하지만 raw 표현공간과 비교하면 garmageNet 1/3641 압축, 우리는 1/64 압축
- 압축률 - 생성품질 tradeoff 는 이미 잘 알려진 사실임. 우리는 일반적인 image LDM 수준의 압축률로 빠른 속도와 더불어서 좋은 성능을 보여주지만 GarmageNet 은 극단적인 압축을 수행함으로써 output quality 가 좋지 않음


### 찐결론

| Method | Raw / mesh face | Compression rate | Latency |
|---|---|---|---|
| GarmageNet | ~30× | 1/3641 | 0.40 s |
| DiffGI | ~1× | 1/64 | 0.80 s |

GarmageNet 의 빠른 inference 는 panel-wise 설계의 자연스러운 귀결이다. 각 panel 을 독립 256x256 image 로 rasterize 하는 구조상 raw 표현공간이 의상 mesh 정보량 대비 약 30 배 더 크고, 이를 transformer 의 고정된 token 길이 안에 담기 위해 1/3641 의 압축비가 적용된다. 이는 image LDM 분야의 표준 압축비 1/50 보다 약 70 배 높은 수치다. DiffGI 는 multi-chart UV packing 으로 의상 전체를 한 장의 256x256 image 안에 담는 구조라, image LDM 표준 범위인 1/64 압축만으로도 동등한 표현 공간을 확보한다. 두 모델의 inference latency 0.40 s 와 0.80 s 는 모두 sub-second 영역이라 실사용 환경에서 차이가 크지 않으며, GarmageNet 의 속도 우위는 efficiency 자체라기보다 표현 압축률 trade-off 의 결과에 가깝다.


### 원본
GarmageNet은 dense mesh를 패널당 72-dim으로 극단적으로 압축하여 빠른 diffusion을 얻지만, 그만큼 품질 측면의 trade-off가 동반됩니다. reconstruction CD on GarmageSet은 $4.7\times$ 악화되고 (본 논문 Tab.~1), image-to-3D BCD는 $\sim\!2\times$ 악화됩니다 (Tab.~5). 본 논문 Fig.~6의 image-to-3D 결과에서도 garment type 변경, sleeve 불일치, 형체 결여 등 row별 실패가 관측됩니다. 또한 fixed per-panel 구조는 패널 수가 가변인 ABO 등 도메인을 지원하지 않습니다 (l.~274–275). 본 방법은 mesh를 과도하게 압축하지 않으면서도 단일 GPU $\sim 1$s latency와 더 높은 품질·일반성을 달성하여 이 trade-off를 보다 효과적으로 해결합니다.


## 수정
GarmageNet은 DiffGI 대비 절반 수준의 콤팩트한 latent space를 구축하여 매우 빠른 생성 속도를 확보했습니다. 하지만 해당 방식은 단일 의상을 다수의 geometry image와 metadata로 정의하기 때문에, raw representation의 크기가 일반적인 multi-chart 방식보다 약 30배 더 방대합니다. 이로 인해 정보의 공격적인 압축(Aggressive compression) 과정에서 치명적인 trade-off가 발생하며, 이는 수치적 성능(Table 1, 5)과 정성적 결과(Figure 6) 모두에서 의상 타입이 왜곡되는 등의 한계로 나타납니다. 두 방법 모두 1초 미만(sub-second)의 낮은 지연 시간을 보이지만, 본 모델은 이러한 제약 속에서도 이미지 조건의 주름 디테일 등을 정교하게 복원하며 우수한 성능을 입증했습니다.

| GarmageNet | 32 × (256 × 256 × 4 + 8) | 32 × 72 | 1/3641 |
| Ours | 256 × 256 × 4 | 32 × 32 × 4 | 1/64 |
